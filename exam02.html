Section 1: ILP & Power Wall (30%)

Understanding the limits of single-core performance and the power constraints that forced the shift to multicore.

Q1 8 Marks

Explain what happened around 2002 in processor design and why it led to multi-core adoption.

  1. Describe the performance trends before 2002.
  2. What changed around 2002?
  3. Why did the industry shift to multi-core?
Technical Answer

a) Trends Before 2002:

  • Exponential Growth: ~50% annual performance improvement.
  • Drivers: Frequency Scaling (MHz -> GHz) and ILP extraction (Instruction Level Parallelism).

b) What Changed (2002):

  • Features hit a wall: Frequency stalled at ~4GHz.
  • The Power Wall: Power density increased to unsustainable levels (nuclear reactor core heat!).
  • ILP limit: Diminishing returns on extracting more parallelism from single threads.

c) Shift to Multi-core:

  • Solution: Instead of one faster, hotter core, use two simpler, slower cores.
  • Benefit: Improves throughput without hitting the power wall (Moore's Law kept giving more transistors).

Q3 14 Marks

Amdahl's Law Calculations (Usually the trickiest part!)

Speedup = 1 / (f + (1-f)/N)

Given: A program where 15% (f=0.15) is sequential.

  1. Calculate speedup on 8 cores.
  2. Calculate maximum theoretical speedup (Infinite cores).
  3. Explain why efficiency decreases as N increases.
Technical Answer

f = 0.15, 1-f = 0.85

a) Speedup (8 Cores):

  • $S = 1 / (0.15 + (0.85/8))$
  • $S = 1 / (0.15 + 0.106) = 1 / 0.256 \approx 3.9$

b) Max Speedup (N -> $\infty$):

  • $S_{max} = 1 / f = 1 / 0.15 \approx 6.67$

c) Efficiency Decrease:

  • As $N$ increases, the parallel portion time shrinks towards zero, but the sequential portion ($f$) remains constant. Most cores end up "waiting" for the sequential part to finish, leading to poor utilization.

Section 3: Memory Systems (25%)

Latency vs Bandwidth, and how Multithreading hides the delays.

Q4 10 Marks

Define and distinguish between Memory Latency and Memory Bandwidth. Give an example for each and explain typical bottlenecks.

Technical Answer
  • Memory Latency: Time delay between the request and the arrival of the first byte. (e.g., 100ns).
    Bottleneck: Pointer chasing, random access.
  • Memory Bandwidth: The rate of data transfer after the first byte arrives. (e.g., 20 GB/s).
    Bottleneck: Large array processing, streaming data.

Key Difference: Latency is "time", Bandwidth is "volume".

Q5 8 Marks

Explain how multi-threading hides memory latency. Compare "Many Small Contexts" (GPU) vs "Few Large Contexts" (CPU).

Technical Answer

Hiding Latency:

  • When one thread stalls waiting for memory (Latency), the hardware instantly switches to another ready thread.
  • This keeps the execution units busy, masking the delay.

Comparison:

  • Many Small Contexts (GPU): Thousands of threads with small register sets. Excellent for hiding latency through brute force parallelism.
  • Few Large Contexts (CPU): 2-4 threads per core with large register sets. Better for complex single-thread performance but less effective at hiding long latencies.

Section 4: Coherence vs Divergence (25%)

CRITICAL: Understanding SIMD execution logic.

Q6 18 Marks

CRITICAL: Explain Coherent Execution vs Divergent Execution in SIMD systems.

âš ï¸ IMPORTANT: This is about instruction stream coherence, NOT cache coherence!
  1. Define Coherent Execution. Why is it efficient?
  2. Define Divergent Execution.
  3. Using the mask example (TTFTFFFF), explain the performance penalty.
Technical Answer

a) Coherent Execution:

  • All SIMD lanes execute the same instruction at the same time.
  • Efficient because one Control Unit manages many ALUs (Amortized Control Overhead). Utilization is 100%.

b) Divergent Execution:

  • Occurs in conditional blocks (If-Else) where threads in the same warp/group disagree on the path.
  • Some threads want to do "A", others want to do "B".

c) Penalty (Masking):

  • SIMD cannot do A and B simultaneously.
  • It serializes: First, it executes "A" for True-threads (False-threads are idle/masked). Then, it executes "B" for False-threads (True-threads idle).
  • Result: Utilization allows drops to 1/N in worst case. The hardware capability is wasted on idle threads.
